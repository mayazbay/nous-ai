# Musk algorithm and Spec-Kit enforcement design

## What this deletes / replaces

- Replaces vague plan-first prose with mandatory six-phase Spec-Kit shape for new meaningful specs.
- Replaces hidden AI copy-paste execution with `ship + learn` proof expectations.
- Does not create a new doctrine skill; updates existing skills only.

## Constitution

- Musk order is requirement, delete, simplify, accelerate, automate.
- Karpathy rule is think before coding, keep it simple, make surgical changes, verify the goal.
- Garry Tan rule is ship what users want and kill what nobody uses.
- Nate B. Jones/OpenBrain rule is ship and learn as separate metrics; avoid cognitive debt.
- RULE ZERO applies: SKILL.md plus gbrain timeline, no new lesson files.

## Specify

Every meaningful plan/spec must include Spec-Kit headers and an explicit deletion section before implementation. Every factory execution closeout must record what shipped and what was learned. Enforcement should start as tests and pre-commit checks, then become mandatory after the canary proves low false positives.

## Clarify

- For this implementation slice, doctrine is codified and tests can be added if already supported by existing gates.
- Broad pre-commit blocking for all historic plan/spec files is risky because old files predate the rule; enforcement should target new files or changed files.

## Plan

- Add doctrine to existing skills rather than creating a new skill.
- Prefer a changed-file detector over scanning the entire vault.
- Require the six headers: `Constitution`, `Specify`, `Clarify`, `Plan`, `Tasks`, `Implement`.
- Require `What this deletes / replaces` in new or edited plans/specs.
